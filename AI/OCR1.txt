# Lucia OCR 기반 복약관리 약품 인식 파이프라인 개선 보고서

## 1. 보고서 목적

본 보고서는 SilverLink AI 복약관리 기능에서 **Lucia OCR이 confidence 없이 텍스트만 반환하는 상황**을 전제로, 기존 OCR → MySQL 후보 검색 → RAG/LLM 설명 구조를 개선하기 위한 새로운 파이프라인을 제안한다.

기존 설계의 핵심 원칙은 유지한다.

```text
LLM은 약을 맞히는 역할이 아니라,
검증된 약품 정보를 사용자가 이해하기 쉽게 설명하는 역할이다.
```

복약관리 기능에서는 LLM이 약품 식별, 처방 변경, 복용 중단 판단, 새로운 의학적 조언 생성을 하면 안 되며, MySQL 기반 후보 검색과 규칙 검증이 중심이 되어야 한다. 기존 설계 문서에서도 Vector DB는 낮은 점수나 애매한 OCR 결과를 보완하는 fallback으로 제한하는 방향을 제시하고 있다. 

---

## 2. 현재 문제 정의

현재 사용 중인 Lucia OCR은 **문자열만 반환**한다.

즉, 다음과 같은 OCR confidence 정보를 사용할 수 없다.

```json
{
  "text": "아모디핀정 5mg",
  "confidence": 0.92
}
```

따라서 기존에 고려했던 **confidence-aware OCR 오류 탐지**는 직접 적용하기 어렵다.

대신 다음 방식으로 전환해야 한다.

```text
OCR confidence-aware 오류 탐지
↓
검색·엔티티·규칙 기반 pseudo-confidence 평가
```

즉 OCR 자체가 제공하지 않는 신뢰도 점수를, 시스템 내부에서 다음 요소를 조합해 계산한다.

```text
- MySQL exact match 여부
- alias/synonym 일치 여부
- synthetic OCR-error alias 일치 여부
- prefix/ngram/fuzzy 유사도
- 함량/단위/숫자 패턴 검증 결과
- 성분/제형/업체 충돌 여부
- top1 후보와 top2 후보의 점수 차이
```

## 3. 최신 연구 및 실무 관점 검토

### 3.1 OCR 결과를 그대로 RAG에 넣는 것은 위험함

최근 OCR-RAG 관련 연구에서는 OCR 노이즈가 RAG 검색과 생성 품질에 연쇄적으로 영향을 줄 수 있다고 본다. ICCV 2025 논문 *OCR Hinders RAG*는 OCR 노이즈를 semantic noise와 formatting noise로 구분하고, OCR 오류가 retrieval과 generation 성능을 모두 저하시킬 수 있음을 평가했다. ([CVF Open Access][1])

따라서 복약관리에서는 OCR 텍스트를 바로 RAG에 넣는 구조보다, 먼저 약품명·함량·성분·제형 같은 **엔티티를 안정적으로 추출하고 검증한 뒤** RAG를 보조적으로 사용하는 것이 안전하다.

---

### 3.2 OCR 오류 보정을 위한 error injection 방식은 타당함

OCR 오류 보정 연구에서는 실제 OCR 오류 패턴을 인위적으로 생성해 학습 또는 보정 데이터로 사용하는 접근이 활용된다. REVISE 연구는 OCR 오류를 character, word, structural level에서 다루고, synthetic data generation 전략을 사용해 OCR 오류 보정을 수행하는 프레임워크를 제안했다. ([ACL Anthology][2])

이를 프로젝트에 바로 적용하면, 모델 학습까지 가지 않더라도 다음 방식으로 실무화할 수 있다.

```text
정상 약품명:
타이레놀

예상 OCR 오류 alias:
타0|레놀
E|이레놀
타이레눌
타이레놀5OOmg
타이레놀5 0 0 mg
```

이렇게 예상 오류 패턴을 `drug_alias` 또는 `drug_error_alias`에 미리 넣어두면, 무거운 fuzzy 검색까지 가지 않고 빠르게 후보를 찾을 수 있다.

---

### 3.3 MySQL ngram/FULLTEXT 활용은 가능하지만 설계가 중요함

MySQL은 CJK 언어 처리를 위해 ngram full-text parser를 제공하며, `FULLTEXT ... WITH PARSER ngram` 형태로 사용할 수 있다. 이는 공백 기반 단어 구분이 어려운 한국어·중국어·일본어 검색에 사용할 수 있는 방식이다. ([Oracle Docs][3])

다만 약품명 검색에서는 ngram/FULLTEXT만으로 최종 판정을 하면 안 된다. ngram은 후보를 넓히는 용도이고, 최종 판단은 함량·성분·제형·업체 검증을 거쳐야 한다.

---

### 3.4 Redis는 source of truth가 아니라 검색 가속 계층으로 적합함

Redis의 cache-aside 패턴은 애플리케이션이 캐시와 DB를 직접 관리하면서, 자주 조회되는 데이터를 캐시에 올려 DB 부하를 줄이는 방식이다. Redis 공식 문서에서도 cache-aside는 애플리케이션이 캐시와 데이터베이스 읽기/쓰기를 직접 관리하는 전략으로 설명한다. ([Redis][4])

복약관리에서는 Redis를 정답 저장소로 쓰기보다 다음 용도로 쓰는 것이 적절하다.

```text
- normalized 약품명 lookup
- alias lookup
- synthetic OCR-error alias lookup
- ngram token → 후보 drug_id set
- OCR raw text → 후보 결과 캐시
- 자주 조회되는 drug detail 캐시
```

정답 데이터의 원본은 MySQL에 두고, Redis는 빠른 조회와 캐시 역할로 제한해야 한다.

## 4. 새로운 파이프라인 개요

### 4.1 최종 추천 파이프라인

```text
[Image / Prescription / Medicine Bag]
        ↓
[Lucia OCR]
        ↓
[Raw Text Store]
        ↓
[Text Normalization]
        ↓
[Optional LLM/sLLM Structured Extraction]
        ↓
[Text Validation & Entity Tagging]
        ↓
[Redis Fast Lookup]
        ↓
[MySQL Candidate Search]
        ↓
[Candidate Merge & Scoring]
        ↓
[Entity-weighted Pseudo-Confidence]
        ↓
[Rule Validation]
        ↓
[Decision]
        ↓
[Vector DB / RAG Fallback]
        ↓
[LLM Explanation]
```

핵심은 **OCR confidence가 없으므로 OCR 자체 신뢰도 대신, 검색 결과와 규칙 검증 결과를 이용해 pseudo-confidence를 계산하는 것**이다.

## 5. 단계별 상세 설계

## 5.1 Lucia OCR

Lucia OCR은 현재 텍스트 추출 역할만 수행한다.

```text
입력:
약봉투 이미지 / 처방전 이미지 / 약품 라벨 이미지

출력:
OCR 텍스트 문자열
```

예시:

```text
타이레놀5 0 0 mg식후30분
```

이 단계에서는 OCR confidence가 없으므로, OCR 결과를 바로 신뢰하지 않는다.

---

## 5.2 Raw Text Store

OCR 원문은 반드시 보존한다.

```json
{
  "ocr_raw_text": "타이레놀5 0 0 mg식후30분",
  "image_id": "IMG_001",
  "request_id": "REQ_001",
  "created_at": "2026-04-27T10:30:00"
}
```

주의할 점은 **정규화 과정에서 원문을 삭제하거나 덮어쓰면 안 된다**는 것이다.

원문은 나중에 다음 용도로 필요하다.

```text
- 오류 분석
- 테스트셋 구축
- OCR error alias 생성
- 사용자 확인 화면 표시
- 재처리 및 개선 평가
```

---

## 5.3 Text Normalization

OCR 텍스트를 검색 가능한 형태로 정리한다.

```text
- 공백 정리
- 특수문자 표준화
- 한글/영문 대소문자 정리
- mg, MG, 밀리그램 → mg
- ml, mL, 밀리리터 → ml
- 정, 정제 → 정
- 캡슐, 캅셀 → 캡슐
```

예시:

```text
원문:
타이레놀5 0 0 mg식후30분

정규화 후보:
타이레놀 500mg 식후 30분
```

다만 정규화는 **확정이 아니라 후보 생성**이어야 한다.

```json
{
  "raw_text": "타이레놀5 0 0 mg식후30분",
  "normalized_candidates": [
    "타이레놀 500mg 식후 30분",
    "타이레놀 50mg 식후 30분"
  ],
  "warnings": [
    "DOSAGE_SPACING_SUSPECTED"
  ]
}
```

---

## 5.4 Optional LLM/sLLM Structured Extraction

이 단계는 선택 사항이다.

LLM을 앞단에 배치할 수는 있지만, 역할은 **약품 확정**이 아니라 **구조화 추출 보조**로 제한해야 한다.

입력:

```text
타이레놀5 0 0 mg식후30분
```

출력:

```json
{
  "drug_name_candidates": [
    {
      "value": "타이레놀",
      "evidence": "타이레놀5 0 0 mg"
    }
  ],
  "strength_candidates": [
    {
      "value": "500",
      "unit": "mg",
      "evidence": "5 0 0 mg",
      "uncertain": true
    }
  ],
  "usage_candidates": [
    {
      "value": "식후 30분",
      "evidence": "식후30분"
    }
  ]
}
```

이 단계의 주의점은 다음과 같다.

```text
LLM이 약품명을 확정하면 안 된다.
LLM이 복용법을 새로 생성하면 안 된다.
LLM이 처방 의도를 해석하면 안 된다.
LLM은 후보를 구조화해서 넘기는 역할만 한다.
```

최종 검증은 반드시 MySQL과 Rule Validation에서 수행한다.

---

## 5.5 Text Validation & Entity Tagging

정규식 기반 검증은 삭제가 아니라 **태깅** 중심으로 수행한다.

나쁜 방식:

```text
이상한 단위 제거
숫자 패턴이 이상하면 잘라냄
```

좋은 방식:

```text
이상한 단위 표시
숫자 패턴 의심 표시
후보를 여러 개 생성
원문 보존
```

예시:

```json
{
  "entities": {
    "drug_name_candidate": "타이레놀",
    "strength_candidate": "500",
    "unit_candidate": "mg",
    "usage_candidate": "식후 30분"
  },
  "warnings": [
    "DOSAGE_DIGIT_SPACING",
    "STRENGTH_UNCERTAIN"
  ]
}
```

복약관리에서는 특히 다음 항목을 강하게 검증해야 한다.

```text
- 약품명
- 함량
- 단위
- 제형
- 성분
- 복용 횟수
- 복용 시간
```

---

## 5.6 Redis Fast Lookup

Redis는 빠른 후보 탐색과 캐시 용도로 사용한다.

### Redis에 저장할 key 예시

```text
drug:exact:{normalized_name} → drug_id
drug:alias:{normalized_alias} → drug_id
drug:error_alias:{synthetic_ocr_error} → drug_id
drug:ngram:{token} → set(drug_id)
ocr:candidate:{hash(raw_text)} → candidate_result_json
drug:detail:{drug_id} → drug_summary_json
```

예시:

```text
drug:alias:타이레놀ER → D001
drug:error_alias:타0|레놀 → D001
drug:error_alias:타이레놀5OOmg → D001
drug:ngram:타이 → {D001, D002}
drug:ngram:레놀 → {D001}
```

Redis를 사용하는 이유는 다음과 같다.

```text
- alias 검색을 빠르게 처리
- synthetic OCR error alias를 빠르게 조회
- 같은 OCR 텍스트 재처리 시 DB 부하 감소
- MySQL ngram/fuzzy 부하 완화
```

단, Redis 데이터는 캐시이므로 MySQL과 동기화 정책이 필요하다.

---

## 5.7 MySQL Candidate Search

MySQL은 정답 데이터의 기준 저장소이다.

검색 순서는 다음과 같이 구성한다.

```text
1. exact
2. alias/synonym
3. synthetic error alias
4. prefix
5. FULLTEXT ngram 또는 별도 ngram table
6. 제한된 후보군에만 fuzzy score 계산
```

중요한 점은 **fuzzy를 전체 DB에 직접 수행하지 않는 것**이다.

권장 구조:

```text
exact/alias/prefix로 빠른 후보 탐색
↓
부족하면 ngram으로 후보 20~50개 확보
↓
그 후보에 대해서만 fuzzy score 계산
```

이렇게 하면 Full Table Scan 위험을 줄일 수 있다.

---

## 5.8 Candidate Merge & Scoring

Redis, MySQL, optional LLM structured extraction에서 나온 후보를 병합한다.

```json
{
  "candidate_drug_id": "D001",
  "product_name": "타이레놀정 500mg",
  "source": [
    "redis_error_alias",
    "mysql_ngram",
    "llm_structured_candidate"
  ],
  "raw_score": 78
}
```

후보 병합 시 같은 `drug_id`가 여러 경로에서 발견되면 점수를 높일 수 있다.

예시:

```text
Redis error alias에서 발견됨
+ MySQL ngram에서도 상위 후보
+ LLM이 약품명 후보로 추출
→ 후보 신뢰도 상승
```

---

## 5.9 Entity-weighted Pseudo-Confidence

Lucia OCR은 confidence를 제공하지 않으므로, 후보 신뢰도는 entity별로 계산한다.

### Entity별 가중치 예시

| 항목                       | 가중치 | 설명           |
| ------------------------ | --: | ------------ |
| 제품명 exact 일치             | +25 | 강한 근거        |
| alias 일치                 | +20 | 별칭 근거        |
| synthetic error alias 일치 | +20 | OCR 오류 보정 근거 |
| 성분 일치                    | +25 | 매우 중요        |
| 함량 일치                    | +30 | 복약 안전상 매우 중요 |
| 단위 일치                    | +20 | mg/ml 오류 방지  |
| 제형 일치                    | +10 | 정/캡슐/시럽 구분   |
| 업체 일치                    |  +5 | 보조 근거        |
| top1/top2 점수 차이 큼        | +10 | 확정성 증가       |
| 함량 충돌                    | -60 | 치명적 감점       |
| 성분 충돌                    | -80 | 후보 제외 수준     |
| 숫자/단위 불확실                | -40 | 사용자 확인 필요    |
| top1/top2 점수 차이 작음       | -30 | 모호성 증가       |

핵심은 **문자열 전체 유사도보다 함량·숫자·성분을 더 강하게 보는 것**이다.

예를 들어 다음 두 경우는 위험도가 다르다.

```text
아모디핀정 → 아모디편정
```

이건 OCR 오타 가능성이 있다.

하지만:

```text
5mg → 50mg
500mg → 50mg
```

이건 복약 안전상 치명적일 수 있으므로 강한 페널티를 줘야 한다.

---

## 5.10 Rule Validation

최종 판정 전에는 반드시 규칙 검증을 수행한다.

검증 항목:

```text
- 성분 일치 여부
- 함량 일치 여부
- 단위 일치 여부
- 제형 일치 여부
- 업체 일치 여부
- 약품명은 같지만 함량이 다른 후보 존재 여부
- 식별문자/모양/색상 정보가 있는 경우 일치 여부
```

규칙 예시:

```text
성분 충돌 있음
→ 후보 제외 또는 AMBIGUOUS

함량 충돌 있음
→ NEED_USER_CONFIRMATION

동일 제품명에 5mg / 10mg 후보가 같이 있음
→ NEED_USER_CONFIRMATION

top1/top2 점수 차이가 작음
→ AMBIGUOUS
```

복약관리에서는 점수가 높더라도 함량 충돌이 있으면 `MATCHED`로 보내면 안 된다.

---

## 5.11 Decision

최종 상태값은 기존 설계를 유지한다.

```text
MATCHED
AMBIGUOUS
LOW_CONFIDENCE
NOT_FOUND
NEED_USER_CONFIRMATION
```

판정 기준 예시:

| 조건                | 상태                     |
| ----------------- | ---------------------- |
| score ≥ 90, 충돌 없음 | MATCHED                |
| 75 ≤ score < 90   | NEED_USER_CONFIRMATION |
| 60 ≤ score < 75   | LOW_CONFIDENCE         |
| score < 60        | NOT_FOUND              |
| top1/top2 차이 작음   | AMBIGUOUS              |
| 함량 충돌             | NEED_USER_CONFIRMATION |
| 성분 충돌             | AMBIGUOUS 또는 후보 제외     |

---

## 5.12 Vector DB / RAG Fallback

Vector DB와 RAG는 최종 판정자가 아니다.

적절한 사용 위치:

```text
- MySQL 후보 점수가 낮은 경우
- alias 표현이 부족한 경우
- 제품 설명 기반 유사 검색이 필요한 경우
- 확정된 약품에 대한 설명 근거 검색이 필요한 경우
```

부적절한 사용 위치:

```text
- 약품 최종 결정
- 함량 확정
- 업체 확정
- 처방 변경 판단
- 복용 중단 판단
```

RAG는 **후보 보완 또는 설명 근거 검색**에만 사용한다.

---

## 5.13 LLM Explanation

LLM은 최종적으로 확정된 구조화 데이터만 받아 설명을 생성한다.

입력 예시:

```json
{
  "drug_name": "타이레놀정 500mg",
  "ingredient": "아세트아미노펜",
  "strength": "500mg",
  "dosage_form": "정제",
  "manufacturer": "예시제약",
  "usage": "처방된 복용법에 따름",
  "cautions": [
    "정해진 용량을 초과하지 마세요.",
    "이상 증상이 있으면 의사 또는 약사에게 문의하세요."
  ]
}
```

프롬프트 원칙:

```text
제공된 데이터만 사용한다.
약품을 새로 식별하지 않는다.
처방 변경을 권고하지 않는다.
복용 중단을 판단하지 않는다.
데이터에 없는 부작용, 효능, 복용법을 생성하지 않는다.
불확실한 내용은 의사 또는 약사에게 확인하도록 안내한다.
```

## 7. Redis 적용 평가

## 7.1 Redis 사용 장점

| 항목                       | 효과                            |
| ------------------------ | ----------------------------- |
| alias 조회 가속              | 자주 쓰는 별칭을 빠르게 조회              |
| synthetic error alias 조회 | OCR 오류 패턴을 빠르게 매칭             |
| MySQL 부하 감소              | 반복 OCR 결과, ngram 후보 조회 캐싱     |
| 응답 속도 개선                 | exact/alias 단계에서 빠르게 후보 결정 가능 |
| 확장성                      | 읽기 중심 workload에 유리            |

## 7.2 Redis 사용 시 주의점

| 위험                   | 대응                         |
| -------------------- | -------------------------- |
| MySQL과 Redis 데이터 불일치 | drug master 변경 시 캐시 무효화    |
| Redis를 정답 DB처럼 사용    | MySQL을 source of truth로 유지 |
| 오래된 OCR 후보 캐시        | TTL 설정                     |
| error alias 과다 등록    | alias 품질 관리 및 충돌 검증        |
| 메모리 사용량 증가           | key 설계와 만료 정책 필요           |

## 7.3 Redis TTL 권장

| 데이터                 | TTL                   |
| ------------------- | --------------------- |
| exact/alias cache   | 장기 또는 수동 무효화          |
| error alias cache   | 장기 또는 배포 시 갱신         |
| OCR raw text 후보 결과  | 1시간~1일                |
| drug detail summary | 1일 또는 master 변경 시 무효화 |
| 사용자 세션성 결과          | 짧은 TTL                |

## 10. 결론

Lucia OCR이 텍스트만 반환하는 현재 조건에서는 **OCR confidence-aware 오류 탐지**를 직접 구현할 수 없다. 따라서 새로운 파이프라인은 OCR confidence 대신 **검색 결과, 엔티티 검증, 후보 점수, 규칙 검증을 조합한 pseudo-confidence 기반 구조**로 설계해야 한다.

최종 권장 방향은 다음과 같다.

```text
Lucia OCR
↓
원문 보존
↓
정규화 및 엔티티 태깅
↓
Redis fast lookup
↓
MySQL 후보 검색
↓
entity-weighted pseudo-confidence
↓
규칙 검증
↓
필요 시 RAG fallback
↓
확정 데이터 기반 LLM 설명
```

이 구조의 핵심은 다음 세 가지이다.

```text
1. MySQL은 source of truth로 유지한다.
2. Redis는 alias, error alias, ngram, OCR 결과 캐시로 검색을 가속한다.
3. LLM과 RAG는 최종 판정자가 아니라 구조화 보조와 설명 생성 역할로 제한한다.
```

따라서 복약관리 기능에서 가장 안전하고 현실적인 설계는 다음과 같이 정리할 수 있다.

```text
OCR 결과가 불완전하다는 전제를 받아들이고,
Redis + MySQL + Entity-weighted Pseudo-Confidence + Rule Validation으로 약품 후보를 검증한 뒤,
LLM은 검증된 정보만 사용자에게 설명하도록 제한하는 구조
```

이 방식이 현재 Lucia OCR 제약, MySQL 성능 우려, 복약관리 안전성, RAG/LLM 활용성을 모두 고려한 가장 적절한 개선 파이프라인이다.
