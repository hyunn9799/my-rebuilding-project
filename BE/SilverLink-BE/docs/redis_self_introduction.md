# Redis 고도화 — 자기소개서 (기술 어필)

---

## 1. Redis를 단순 캐시에서 실시간 인프라로 전환한 경험

저는 SilverLink AI 프로젝트에서 Redis를 **"키-값 저장소"를 넘어 "실시간 세션 보안 인프라"**로 고도화한 경험이 있습니다.

프로젝트 초기의 Redis 구현은 기본적인 세션 저장 용도로만 사용되고 있었습니다. 2026년 실무 기준으로 평가한 결과, 11가지 평가 영역에서 **42/110점(38%)**에 불과했습니다. CacheManager Bean이 등록되어 있었지만 실제로 `@Cacheable`을 사용하는 곳이 한 곳도 없었고, 헬스체크 코드에서 **매번 커넥션을 열고 닫지 않는** 치명적 누수 버그가 존재했으며, Redis 장애 시 전체 API가 다운되는 구조적 문제가 있었습니다.

저는 이 문제를 **6단계 로드맵으로 체계화**하고, 단순 수정이 아닌 **산업 표준 수준의 아키텍처 설계**로 접근했습니다.

## 2. 기술적으로 가장 깊이 고민한 부분

### (1) Lua Script로 Race Condition 원천 차단

동시 로그인 처리에서 가장 어려웠던 문제는 **"기존 세션 무효화"와 "새 세션 생성" 사이의 간극**이었습니다. 두 요청이 동시에 들어오면of, 기존 세션이 무효화되지 않은 채 새 세션이 생기는 좀비 세션이 발생했습니다.

`MULTI/EXEC` 트랜잭션은 "조회 후 판단 후 쓰기" 패턴에서 Check-and-Set이 불가능했기에, **Lua Script를 설계하여 서버 사이드에서 단일 원자적 명령으로 실행**되도록 구현했습니다. 이는 실무에서 **Redis의 단일 스레드 특성을 활용한 동시성 제어**의 정석적 접근입니다.

### (2) Circuit Breaker + 3계층 Fallback으로 완전한 장애 격리

"Redis가 죽으면 어떻게 할 것인가?"라는 질문에 단순히 예외를 잡는 것이 아니라, **Resilience4j Circuit Breaker + 3계층 Fallback Cache (Redis → LocalConcurrentMap → DB)**를 조합하여 설계했습니다.

특히 `isActive()` 메서드의 Fallback에서 `return true`가 아닌 **`return false`**를 선택한 것은 중요한 설계 판단이었습니다. 가용성(Availability)보다 **보안(Security)을 우선**하여, Redis 장애 시 인증되지 않은 요청이 통과되는 것을 원천 차단했습니다.

### (3) Pipeline으로 정량적 성능 개선

`touch()` 메서드는 매 API 요청마다 호출되는 고빈도 메서드입니다. 기존에는 hSet → expire → expire로 **3회 네트워크 왕복**이 발생했는데, Redis Pipeline으로 배치 처리하여 **1회 왕복으로 66% 감소**시켰습니다. `invalidateBySid()`에서도 동일한 최적화를 적용했습니다.

네트워크 레이턴시가 1ms라 가정해도, 초당 1000 TPS 환경에서 **2ms × 1000 = 2초**의 누적 지연이 절감되는 효과입니다.

## 3. Pub/Sub과 Keyspace Notification으로 실시간 아키텍처 구현

SilverLink는 독거노인 돌봄 플랫폼이라 **관리자가 여러 기기에서 동시 접속하는 시나리오**가 빈번합니다. "1계정 1기기" 정책을 강제하려면, 세션이 강제 종료되었을 때 **다른 서버 인스턴스에도 즉시 전파**되어야 했습니다.

Redis Pub/Sub을 통해 세션 킥 메시지를 실시간으로 발행하고, 모든 인스턴스가 구독하여 일관된 세션 관리를 구현했습니다. 동시에 Redis Keyspace Notification을 활용하여 TTL 만료된 세션에 대한 자동 후처리(감사 로그, 통계 집계)도 수행합니다.

## 4. 감사 로그를 "운영 자산"으로 전환

기존에는 보안 이벤트가 `log.info()`로만 출력되어, 로그 파일을 grep해야만 확인할 수 있었습니다. 이를 **Spring Event + @Async + JPA**를 조합하여 DB에 영속화하고, **REST API로 관리자 대시보드에서 실시간 조회**할 수 있도록 개선했습니다.

특히 `@Async`로 비동기 처리하여 **감사 로그 저장이 API 응답 시간에 영향을 주지 않도록** 설계한 점, 그리고 DB 저장 실패 시에도 **로그 출력은 완료된 상태이므로 데이터 유실을 방지**하는 방어적 코딩을 적용했습니다.

---

## 5. 테스트 전략: Mock을 넘어 Testcontainers로

단위 테스트 19개는 Mockito로 검증했지만, **Lua Script의 원자적 실행, Pipeline의 순서 보장, TTL 만료 동작** 등은 Mock으로는 검증할 수 없었습니다. 

**Testcontainers로 실제 Redis 7 컨테이너를 띄우고 8개 통합 테스트**를 작성하여, Lua Script Atomic Swap, Pipeline TTL 갱신, HMGET 다중 필드 조회, 리프레시 토큰 회전 등을 실제 Redis 환경에서 검증했습니다.

## 6. 이 경험을 통해 얻은 것

| 관점 | 배운 점 |
|------|---------|
| **설계** | "도구를 쓰는 것"과 "도구를 제대로 쓰는 것"의 차이를 깨달았습니다 |
| **성능** | Pipeline과 Batch의 중요성을 정량적으로 입증할 수 있게 되었습니다 |
| **안정성** | "장애가 발생하면 어떻게 되는가?"를 항상 먼저 생각하는 습관을 갖게 되었습니다 |
| **보안** | Availability vs Security 트레이드오프에서 올바른 선택을 내리는 판단력을 키웠습니다 |
| **테스트** | Mock의 한계를 인식하고 통합 테스트의 필요성을 체감했습니다 |

Redis를 "캐시"가 아닌 **"실시간 분산 인프라"**로 활용할 줄 아는 개발자로 성장했으며, 이러한 경험을 바탕으로 실무에서도 안정적이고 확장 가능한 시스템을 설계하고 구현할 수 있다고 확신합니다.
